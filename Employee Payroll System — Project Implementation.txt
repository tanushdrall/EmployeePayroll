# Employee Payroll System — Project Implementation

## How I approached this project

Before writing any code I thought about what the app needed and how to structure it.

---

## Step 1 — Define Entities

Created separate files for each data class:

```
Employee.cs    →  Id, Name, DepartmentId, BaseSalary
Department.cs  →  Id, Name
Salary.cs      →  Id, EmployeeId, GrossSalary, Deductions, NetSalary, PayDate, ProcessedBy
```

Each entity is a simple data class with no business logic.
This follows Single Responsibility — data only.

---

## Step 2 — Define Interfaces

Before writing implementations, defined contracts:

```
IEmployeeRepository  →  Add, Delete, Update, GetById, GetAll (all async)
ISalaryRepository    →  SaveAsync
ISalaryCalculator    →  Calculate
IValidator<T>        →  Validate
IPayrollLogger       →  Log
```

Interfaces first means depending on abstraction not implementation.
This is Dependency Inversion applied from the start.

---

## Step 3 — Build Services

Services orchestrate the business logic:

```
EmployeeService  →  validates then adds employee, gets all employees
PayrollService   →  fetches employees, validates, calculates, saves salary
```

Services depend on interfaces only — no concrete classes in constructors.

---

## Step 4 — Implement Concrete Classes

Actual working implementations:

```
EmployeeRepository   →  async EF Core operations, ILogger for error tracking
SalaryRepository     →  saves calculated salary to DB
SalaryCalculator     →  10% tax deduction, returns Salary object
EmployeeValidator    →  checks Name, BaseSalary, DepartmentId
ConsolePayrollLogger →  logs to console with timestamp
```

Each class implements one interface and does one job.

---

## Step 5 — Wire DI Container

Registered all services in Program.cs:

```csharp
services.AddDbContext<PayrollContext>(options =>
    options.UseSqlServer(config.GetConnectionString("PayrollDb")));

services.AddLogging(builder => builder.AddConsole());
services.AddTransient<IEmployeeRepository, EmployeeRepository>();
services.AddTransient<IValidator<Employee>, EmployeeValidator>();
services.AddTransient<ISalaryCalculator, SalaryCalculator>();
services.AddTransient<ISalaryRepository, SalaryRepository>();
services.AddSingleton<IPayrollLogger, ConsolePayrollILogger>();
services.AddTransient<EmployeeService>();
services.AddTransient<PayrollService>();
```

Container resolves all dependencies automatically.

---

## Step 6 — Console Menu

Built interactive menu for end-to-end usage:

```
1. Add Employee    →  reads input, validates, saves to DB
2. Process Payroll →  fetches all, calculates, saves salary
3. View Employees  →  displays all from DB
4. Exit            →  shuts down safely
```

---

## Step 7 — Unit Tests

Written for critical business logic:

```
EmployeeValidatorTests  →  7 cases covering valid, empty name,
                           null name, zero salary, negative salary,
                           invalid department, boundary salary

SalaryCalculatorTests   →  verifies gross and net calculation correct
```

---

## Step 8 — EF Core and Migrations

```
PayrollContext  →  DbSet for Employee, Department, Salary
Migrations      →  InitialCreate, AddSalaryTable, FixBaseSalaryPrecision
Seeding         →  3 departments, 3 employees via HasData
Precision       →  HasColumnType decimal(18,2) for salary fields
```

---

## Step 9 — Configuration

Moved connection string from hardcoded to appsettings.json:

```json
{
  "ConnectionStrings": {
    "PayrollDb": "Server=...;Database=PayrollDb;..."
  }
}
```

appsettings.json excluded from source control via .gitignore.

---

## Key lessons from building this app

```
IDENTITY_INSERT      →  never set Id manually, DB generates it
rethrow after log    →  caller must know when operation failed
one foreach loop     →  validate and process in same iteration
decimal precision    →  always specify HasColumnType for decimal
BaseSalary cast      →  never cast decimal to int, data lost
appsettings.json     →  connection string out of source code
ILogger<T>           →  use built-in not custom for repositories
AddLogging required  →  ILogger<T> needs explicit registration
```

---

## What the app does not have

```
→  Authentication and authorization
→  Role based access control
→  Web or API layer
→  Department validation against DB on add
→  Update and delete employee via menu
→  CancellationToken in async methods
→  Integration tests
```

---

## Honest reflection

The app took longer than expected because gaps kept appearing as new features were added. The pattern of fixing one thing and breaking another showed why building clean foundations matters from the start.

SOLID and DI are not just principles — they are the reason this app can be tested, extended, and maintained. Without them the app would have been a God class in Program.cs with no tests and no structure.
